slackwire

A command-line tool for sending, updating, and managing rich Slack messages, from your terminal or CI.

v0.1.0 · requires Node.js 20+

Quickstart · 30 seconds
sh
# 1. point slackwire at a token (bot or user)
export SLACK_TOKEN=xoxb-...

# 2. post a message
npx slackwire post --channel C0XXXXXXXXX --text "hello from slackwire"

On success, post prints the message timestamp and a permalink, tab-separated:

stdout
1718900000.123456	https://slack.com/archives/C0XXXXXXXXX/p1718900000123456

Capture that ts and you can update, react to, or delete the same message later. That is the whole idea.

Why slackwire

Most Slack notification tools are send-only webhook wrappers: they post a message and nothing more. slackwire talks to the Slack Web API, so it can post a message and then live-update the same one, morphing its status, text, and accent color in place.

Post and live-morph

Post a running card, capture its ts, then morph it to passed or failed on the same message. Also delete, react, and upload.

Never corrupts

Per-kind escaping and structural limit guards mean < > &, values like Name <email>, and oversized payloads fail loudly instead of silently breaking the card.

Built for CI

Predictable exit codes, a token-from-file pattern for secrets, and a non-blocking fail mode so a notification step never breaks your pipeline.

Templates are optional. Most usage is a single --text or --blocks call; reach for templates only when you post the same shaped card repeatedly.

Install

slackwire ships on three rails. Pick whichever fits your environment. All three run the identical bundled CLI.

1. Bare bundled binary

A single self-contained CommonJS file, no install step. Useful when you already have the repo checked out.

sh
node packages/cli/dist/bundle.cjs post --channel C0XXXXXXXXX --text "hi"

2. Global npm install or npx

The published package is the unscoped name slackwire with a slackwire bin.

sh
npm install -g slackwire
slackwire post --channel C0XXXXXXXXX --text "hi"

# or without installing anything
npx slackwire post --channel C0XXXXXXXXX --text "hi"

3. Distroless Docker image

Built from the included Dockerfile (a distroless nodejs20 base with only the bundled CLI inside).

sh
docker build -t slackwire:dev .
docker run --rm -e SLACK_TOKEN slackwire:dev post --channel C0XXXXXXXXX --text "hi"
Note

The image contains only the CLI, not the templates/ catalog. To use templates in a container, mount your catalog and point SLACK_CATALOG (or --catalog) at the mount. See Templates.

Authentication & configuration

The token is read from the environment, never from argv, and is never logged. Errors that would echo a token have it redacted. The token is resolved in this order, first match wins:

OrderSourceBehavior
1SLACK_TOKENUsed directly. Highest precedence.
2SLACK_TOKEN_BASE64Base64-decoded at runtime. Used if SLACK_TOKEN is unset.
3SLACK_TOKEN_FILEPath to a file holding the token; trimmed file contents are used. The Docker-secret pattern. Used if the two above are unset.

A bot token (xoxb-...) and a user token (xoxp-...) both work. The choice matters for cross-pipeline re-find behavior, see Using it in CI.

Environment variable matrix

VariablePurposeDefault
SLACK_TOKENSlack token, highest precedence.none
SLACK_TOKEN_BASE64Base64-encoded token, used if SLACK_TOKEN is unset.none
SLACK_TOKEN_FILEPath to a token file, used if the two above are unset.none
SLACK_ATTRIBUTIONSet to true to render the colored accent bar (rides on a legacy attachment). See the caveat.off
SLACK_CATALOGTemplate catalog directory../templates
SLACK_TEAM_IDTeam ID used when resolving channel and user names.T000
HTTPS_PROXY / NO_PROXYStandard proxy passthrough for egress-restricted runners.none
Tip

The SLACK_TOKEN_FILE pattern pairs cleanly with Docker and Kubernetes secrets: mount the secret as a file, set SLACK_TOKEN_FILE to its path, and the token never appears in your shell history or process list.

Commands

slackwire is invoked as slackwire <verb> [options]. Running it with no verb prints usage and exits 2; an unknown verb also exits 2.

VerbShapePrints on success
post--channel <c> with one of --template / --blocks / --text<ts>\t<permalink>
update--channel <c> --ts <ts> with optional body source<ts>\t<permalink>
delete--channel <c> --ts <ts>deleted\t<ts>
react--channel <c> --ts <ts> --emoji <name>reacted\t<emoji>
upload--channel <c> --file <path> [--title] [--comment]uploaded\t<path>
card--template <n@v> --channel <c> [--data] — alias for post --template<ts>\t<permalink>

For post and update, the message body comes from exactly one of three sources: --template, --blocks, or --text. (For update, the body is optional: update --channel --ts with no body still issues a chat.update call.)

Flag reference

FlagMeaning

--template <name@ver>Template to render. Version defaults to 1.0.0.
--data <json>JSON payload for the template. Invalid JSON exits 2.

--emoji <name>Reaction emoji name, without colons.
--file <path>File to upload.
--title <t> / --comment <c>File title and initial comment for upload.
--catalog <path>Template catalog directory. Overrides SLACK_CATALOG (default ./templates).

post · plain text

sh
slackwire post --channel C0XXXXXXXXX --text "deploy finished"

--channel accepts a channel ID or a channel name. Names are resolved to an ID via the Slack API before posting.

post · raw Block Kit

--blocks takes a JSON array of blocks, or an object shaped { "blocks": [...], "attachments": [...], "text": "..." }. slackwire validates structure and Slack limits before sending, and auto-derives the fallback text from the blocks if you do not supply --text.

sh
slackwire post --channel C0XXXXXXXXX \
  --blocks '[{"type":"section","text":{"type":"mrkdwn","text":"*Build* passed"}}]'

Pass --blocks - to read the JSON from stdin:

sh
cat card.json | slackwire post --channel C0XXXXXXXXX --blocks -

post · from a template

Templates render typed placeholders into Block Kit. Provide the template as name@version (version defaults to 1.0.0) and feed it data with --data.

sh
slackwire post --channel C0XXXXXXXXX \
  --template ci-cd@1.0.0 \
  --data '{"title":"CI passed","ref":"main","author":"Naledi"}'

card is an alias for post --template:

sh
slackwire card --template ci-cd@1.0.0 --channel C0XXXXXXXXX --data '{...}'

Add --theme '#2eb67d' to set an accent color (see the caveat). The catalog defaults to ./templates and is overridden with --catalog <path> or SLACK_CATALOG.

update · live-morph

update rewrites a message in place by its timestamp. The canonical pattern is to post a running card, capture its ts, then morph it to passed or failed when the job finishes. This is what makes a single Slack message track a job from start to finish instead of spamming the channel.

sh
# post the running card, capture the timestamp (first tab-separated field)
TS=$(slackwire post --channel C0XXXXXXXXX --template ci-cd@1.0.0 \
       --data '{"status":"running"}' | cut -f1)

# later, morph the same message to passed
slackwire update --channel C0XXXXXXXXX --ts "$TS" --template ci-cd@1.0.0 \
  --data '{"status":"passed"}'

To morph the accent color along with the status, pass --theme and set SLACK_ATTRIBUTION=true: a green bar on passed, a red bar on failed. update also prints <ts>\t<permalink> on success.

delete

sh
slackwire delete --channel C0XXXXXXXXX --ts 1718900000.123456

Prints deleted\t<ts> on success.

react

sh
slackwire react --channel C0XXXXXXXXX --ts 1718900000.123456 --emoji white_check_mark

Use the emoji name without colons. Prints reacted\t<emoji> on success.

upload

sh
slackwire upload --channel C0XXXXXXXXX --file ./report.txt \
  --title "Coverage report" --comment "Latest run"

--title and --comment (Slack's initial comment) are optional. Prints uploaded\t<path> on success.

--dry-run

--dry-run renders the message from --template / --blocks / --text and prints the assembled JSON ({blocks, text, attachments?}) to stdout, without a token and without calling Slack. Use it to preview a message or validate a template in CI.

sh
slackwire post --channel C0XXXXXXXXX --blocks '[{"type":"section","text":{"type":"mrkdwn","text":"*Build* passed"}}]' --dry-run
stdout
{
  "text": "Build passed",
  "blocks": [
    {
      "type": "section",
      "text": { "type": "mrkdwn", "text": "*Build* passed" }
    }
  ]
}

The fallback text is auto-derived from the block content. Validation still runs, so a template missing a required field or a block over a Slack limit exits 2 even in dry-run.

--fail-mode

--fail-mode controls what happens when a Slack call fails. It defaults to non-blocking so a notification step never breaks your pipeline. See Exit codes for the full contract.

sh
slackwire post --channel C0XXXXXXXXX --text "hi" --fail-mode block

--theme

--theme '#rrggbb' sets the accent color of the card. The colored bar only renders when SLACK_ATTRIBUTION=true, because it rides on a legacy Slack attachment. Read the gotcha before turning it on.

sh
export SLACK_ATTRIBUTION=true
slackwire card --template ci-cd@1.0.0 --channel C0XXXXXXXXX \
  --theme '#2eb67d' --data '{"status":"passed"}'

Templates · optional

Templates are a convenience for cards you post repeatedly, such as CI/CD status or incident updates. You do not need them for ad hoc messages; --text and --blocks cover most usage.

Shape: one directory per card

Each template is a versioned directory holding three files:

FileHolds
skeleton.jsonBlock Kit JSON with {{kind:key}} placeholder tokens.
schema.jsonA map of each field name to its placeholder kind.
meta.jsonName, version, description, and morph states.

Typed placeholders

Placeholders are typed, written {{kind:key}}, and each kind is escaped for its position. A bare untyped {{var}} is not allowed: the engine rejects it with a clear error telling you to use {{kind:path}} syntax. Per-kind escaping is why < > & and values like Name <email> render safely instead of breaking the markup.

KindUsed forEscaping
text_plainHeader / plain_text fields.HTML-escapes & < >.
text_mrkdwnMarkdown section / context text.HTML-escapes & < >.
link_urlButton / link URLs.Parsed and validated; only http / https allowed.
image_urlImage and icon URLs.Same URL validation as link_url.
dateSlack <!date> tokens.Requires integer-seconds epoch; rejects millisecond values.

The engine supports further kinds (link text, code, mentions, color, button) for richer cards; the five above cover the common case.

A concrete template

The bundled ci-cd@1.0.0 template. The schema.json declares every field and its kind:

schema.json
{
  "title": "text_plain",
  "ref": "text_plain",
  "author": "text_mrkdwn",
  "icon_url": "image_url",
  "finished_at": "date",
  "primary_url": "link_url",
  "logs_url": "link_url"
}

The matching skeleton.json references those fields by kind and key. An excerpt:

skeleton.json
{ "type": "header",
  "text": { "type": "plain_text", "text": "{{text_plain:title}}" } },
{ "type": "section",
  "text": { "type": "mrkdwn",
    "text": "`{{text_plain:ref}}` by *{{text_mrkdwn:author}}*" } }

And meta.json names the card and its morph states:

meta.json
{
  "name": "ci-cd",
  "version": "1.0.0",
  "description": "CI/CD pipeline live card with morph states (running/passed/failed)",
  "states": ["running", "passed", "failed"]
}
Tip

Templates are byte-parity tested against hand-authored reference cards, so the render engine cannot silently drift from the cards a designer signed off on. Preview any template with --dry-run before wiring it into CI.

Note

Templates live in your repo catalog (default ./templates), not baked into the Docker image. In a container, mount the catalog and point SLACK_CATALOG or --catalog at the mount.

Using it in CI (GitLab)

slackwire is designed to run inside a pipeline: a running card on the .pre stage, its ts handed forward through a dotenv artifact, and a morph to passed or failed on completion. The Docker image and npx slackwire both work as CI steps.

The live-morph pattern

  1. notify:running posts the initial card and writes the Slack ts to a dotenv artifact.
  2. notify:success reads the ts and calls update to morph the card to passed.
  3. notify:failure does the same on failure, but uses dependencies: instead of needs:.
.gitlab-ci.yml
notify:running:
  stage: .pre
  image: slackwire:dev
  script:
    - slackwire card --template cicd-live@1.0.0 \
        --channel "$SLACK_CHANNEL" \
        --data "{\"status\":\"running\",\"ref\":\"$CI_COMMIT_REF_NAME\"}" | tee slack.env
    - echo "SLACK_TS=$(awk '{print $1}' slack.env)" >> notify.env
  artifacts:
    reports:
      dotenv: notify.env

notify:success:
  stage: .post
  image: slackwire:dev
  needs: [{ job: notify:running, artifacts: true }]
  script:
    - slackwire update --channel "$SLACK_CHANNEL" --ts "$SLACK_TS" \
        --template cicd-live@1.0.0 --data '{"status":"passed"}'
  when: on_success

notify:failure:
  stage: .post
  image: slackwire:dev
  dependencies: [notify:running]   # not needs: (see below)
  script:
    - slackwire update --channel "$SLACK_CHANNEL" --ts "$SLACK_TS" \
        --template cicd-live@1.0.0 --data '{"status":"failed"}'
  when: on_failure
Gotcha · needs: vs dependencies:

On the failure path, use dependencies:, not needs:. When a job fails, any downstream job that declares needs: [failed-job] enters the amber "blocked" state and never runs. dependencies: only controls artifact download, not execution ordering, so the failure notification still fires on when: on_failure.

Idempotency · the dotenv is just a cache

The dotenv ts is a convenience, not the source of truth. If the artifact is lost (a retry, or a separate release pipeline morphing the same card), the CLI re-finds the existing message via a dedupe key stored in Slack message metadata. Set a consistent key across every job that touches the card:

.gitlab-ci.yml
variables:
  SLACK_DEDUPE_KEY: "pipeline-$CI_PIPELINE_ID"

Pass it as dedupeKey in your --data JSON. A bot token re-finds via conversations.history scanning the metadata payload; a user token re-finds via search.messages on the dedupe key.

Egress-restricted runners

For runners without direct internet access, pass standard proxy settings. Include the AWS metadata range in NO_PROXY if your runners run on EC2; include slack.com only if your proxy terminates HTTPS for the Slack API.

.gitlab-ci.yml
variables:
  HTTPS_PROXY: "http://proxy.internal:8080"
  NO_PROXY: "169.254.0.0/16,.internal"

Exit codes & error handling

CodeMeaning
0Success (or a non-blocking Slack / network / rate-limit warning).
2Validation error: bad input, missing required flag, invalid --data / --blocks JSON, schema / structural / limit error, missing token, or unknown verb.
3Slack API error (in --fail-mode block).
4Network or timeout error (in --fail-mode block).
5Rate-limited and gave up (in --fail-mode block).

--fail-mode

  • non-blocking (default): a Slack, network, or rate-limit failure prints a warning to stderr and exits 0, so a notification step never breaks your pipeline.
  • block: those failures exit with the non-zero code from the table above.
Always

Validation errors always exit 2, regardless of fail mode. A malformed card is a bug in your pipeline, not a flaky network, so slackwire never swallows it. A missing token also exits 2, unless --dry-run is set.

How it works

Every templated message runs through a fixed render pipeline. Each stage either transforms the card or refuses it, so a card that reaches Slack has already passed every guard.

render pipeline
  load        read skeleton.json + schema.json + meta.json
   |
   v
 validate     payload checked against a JSON schema built from the kinds
   |
   v
 per-kind     each value escaped for its slot (HTML, URL, date, ...)
 escape
   |
   v
 interpolate  {{kind:key}} tokens replaced; bare {{var}} rejected
   |
   v
 assemble     blocks + partials composed into the final card
   |
   v
 structural   block shapes valid, image blocks have alt_text
 + limit      50 blocks, 3000 / 150 / 75 char caps, ~38000 total
 guards
   |
   v
 fallback     plain-text fallback derived from block content
   |
   v
 emit         { blocks, text, attachments? }

Raw --blocks input skips render and interpolation but still runs through the structural and limit guards and the fallback derivation before sending.

Monorepo layout

The published slackwire CLI is a thin front end over @slackwire/core, the render engine and Slack client. The same core also backs an MCP server.

packages/

  cli/    slackwire         the command-line front end

Limits & gotchas

Slack hard limits

Enforced before sending. Exceeding any of these is a validation error (exit 2), not a silent truncation.

LimitCap
Blocks per message50
Section text3000 characters
Header text150 characters
Button label75 characters
Total payload (soft)~38000 characters
Image blocksrequire alt_text
Gotcha · the colored bar & "Added by"

The colored accent bar uses a legacy Slack attachment. Setting --theme plus SLACK_ATTRIBUTION=true enables it, but because it is a legacy attachment Slack may render an "Added by <app>" footer on the message. A native Alert block is the planned footer-free path.

Gotcha · attachment blocks not validated

attachments[].blocks are not yet structurally validated by the engine. Only top-level blocks are length- and alt_text-checked. If you move a block tree into an attachment (the house-style colored-card pattern), respect Slack's limits yourself.

Note · name resolution

Channel name resolution uses a non-persisting (noop) cache in the CLI. If a name cannot be resolved, the raw string is passed to the API unchanged. Pass a channel ID directly to skip resolution entirely.

MCP server

slackwire has two faces over one core. The CLI is the terminal/CI front end; the Model Context Protocol server, @slackwire/mcp, is the front end for MCP-capable assistants. Both call the same @slackwire/core engine, so escaping, limit guards, and fallback derivation apply identically no matter which face you use.

The package now ships a runnable stdio server, not just a library. Installing it provides a slackwire-mcp bin (entry dist/main.js) that starts a stdio MCP server and resolves the Slack token from the environment, first match wins: SLACK_TOKEN, then SLACK_TOKEN_BASE64 (base64-decoded), then SLACK_TOKEN_FILE (a path to a token file). Diagnostic logs go to stderr, prefixed [slackwire-mcp].

Run it

sh
export SLACK_TOKEN=xoxb-...
slackwire-mcp           # once @slackwire/mcp is installed

# or without installing
npx -y slackwire-mcp

Wire it into an MCP client

Point a Claude Desktop / Claude Code style MCP client at the bin and feed it a token through the environment.

json
{
  "mcpServers": {
    "slackwire": {
      "command": "npx",
      "args": ["-y", "slackwire-mcp"],
      "env": { "SLACK_TOKEN_FILE": "/absolute/path/to/.slack_token" }
    }
  }
}
